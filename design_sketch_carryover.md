8. **Community feedback shaping the design.** The user is engaged with the ICM community. Three design takeaways from that engagement (captured 2026-06-02; specifics not preserved in this doc — see ARCHITECT_DECISIONS for the permanent record):

   - **sem_debug is a "drift alarm," not a "provenance map."** The community consensus (multiple users): a flag saying "this passage needs human inspection" is more useful than a citation system that pretends to know which source each passage came from. The MCP tool's `trace_report` output should be framed as a flag, with the flagged passages attached. The user decides what to do with the flag. This sharpens open question #2: there is no severity ladder. Every flag is "needs inspection." The action is `accept | rerun | rerun_strict | abort`.

   - **Section-loading as input discipline.** Tight, declared input sections shrink the attribution surface and make drift detection more reliable. Our `CONTEXT.md` Inputs table is the contract — declaring which input sections a stage is allowed to read. v1 makes the contract declarative; v2 makes it enforceable (e.g., stage_runner reads only the declared sections from the prior stage's output).

   - **Raw source capture as a future mechanism.** A community-shared example of a fully auditable run: synthesis is append-only, source copies are preserved, run/approval logs are a third input, raw source archived for a retention period. The decision trail format (structured `decisions.md` per stage) supports this pattern. The mechanism for raw source capture is v2; v1's `runs/<uuid>/<stage>/findings.md` is the seed for that mechanism.
